# Gomoku AI 技术文档

这份文档专门解释本项目的五子棋 AI 是怎么工作的。它默认读者已经有一定 C++、数据结构和搜索算法基础，但不要求熟悉棋类 AI。整体目标是讲清楚：程序为什么能下棋、为什么能比较快、为什么大多数情况下不会乱下，以及它还有哪些局限。

如果只想运行程序、查看 exe 名称或编译命令，请看 `README.md`。这份文档更像“算法说明书”。

## 1. 整体架构

项目的核心只有一个源码文件：`main.cpp`。黑方、白方、不同速度档位、显示棋盘版本、随机版本，都是通过编译宏从同一份代码生成的。

例如：

```powershell
g++ -O2 -std=c++17 -DBLACK_AI main.cpp -o black.exe
g++ -O2 -std=c++17 -DWHITE_AI main.cpp -o white.exe
g++ -O2 -std=c++17 -DBLACK_AI -DSUPER_AI main.cpp -o black_super.exe
g++ -O2 -std=c++17 -DBLACK_AI -DRANDOM_AI main.cpp -o random_ai/black_random.exe
```

这样做的好处是：黑白双方使用同一套规则、同一套搜索、同一套评估函数，不会出现“黑方修了一个 bug，白方忘了修”的情况。

### 1.1 程序怎样和外界交互

程序只通过标准输入和标准输出交换坐标。坐标格式是：

```text
row col
```

例如：

```text
15 15
```

外部坐标是 `1..38`，但源码内部数组使用 `0..37`。所以输入时会减 1，输出时会加 1。

黑方程序比较特殊，因为黑棋先走：

1. `black.exe` 启动后先自己下第一手并输出。
2. 之后每次读入白方一步，再输出黑方一步。

白方程序则是：

1. `white.exe` 启动后先等待黑方坐标。
2. 读入黑方一步后，输出白方一步。
3. 后面继续循环。

代码中的主要入口流程可以概括为：

```text
初始化哈希和随机数
如果我是黑方：先选择并输出第一手
循环：
    读取对手坐标
    更新棋盘
    AI 选择自己的下一步
    更新棋盘
    输出坐标
```

## 2. 棋盘和局面表示

棋盘是一个 `38 x 38` 的二维数组：

```cpp
static int board[N][N];
```

每个格子有三种状态：

```cpp
EMPTY = 0
BLACK = 1
WHITE = 2
```

这是一种最直接的表示方法。它的优点是简单，判断某个点有没有棋子、某条线连续几个棋子都很方便。缺点是如果要做特别高性能的棋类 AI，二维数组不如位棋盘等结构极致。但本项目棋盘是 38 路，规则还包含禁手，直接数组更容易写对，也更容易解释。

程序里还有一个 `moveCount`，记录当前已经下了多少手。它用于判断开局、控制候选点半径等。

## 3. AI 的一次决策流程

核心函数是：

```cpp
chooseMove(int color)
```

可以把它理解为“给定当前棋盘和轮到谁下，返回一个坐标”。

它大致做 5 件事：

1. 如果是空棋盘，确定版下中心点，随机版在中心附近随机。
2. 设置本次搜索的时间限制。
3. 生成一批候选点。
4. 先检查有没有一步赢，或者必须挡对方一步赢。
5. 如果没有简单答案，就用 alpha-beta 搜索比较这些候选点。

这个流程很像人下棋：先看有没有“马上赢”或“马上输”的明显情况；如果没有，再分析未来几步。

## 4. 为什么不能全盘搜索

38 路棋盘一共有：

```text
38 * 38 = 1444
```

个交点。假设 AI 每一步都考虑所有空点，搜索 4 层大概是：

```text
1444 * 1443 * 1442 * 1441
```

这是天文数字，普通电脑根本算不完。

但五子棋有一个很重要的特点：有价值的棋通常在已有棋子附近。离所有棋子都很远的空点，绝大多数时候既不能进攻，也不能防守。

所以程序用 `generateCandidates` 生成候选点，只考虑已有棋子周围一圈或几圈的空点：

```cpp
static vector<Candidate> generateCandidates(int color, int limit)
```

开局前几手棋子少，程序使用半径 `3`，给布局留一些空间。中后盘棋子多，使用半径 `2`，把搜索集中在战斗区域。然后每个候选点会被打分排序，只保留前 `limit` 个。

这个步骤非常关键。它相当于告诉搜索算法：“先别管全世界，只看最可能有意义的地方。”

## 5. 候选点的局部评分

候选点排序由 `pointShapeScore` 完成：

```cpp
static int pointShapeScore(int r, int c, int color)
```

它不是最终判断谁赢谁输的评估函数，而是给候选点排序用的“直觉分”。比如一个点如果下了可以直接五连，它就非常重要；如果一个点可以挡住对手五连，也非常重要；如果只是离中心近，则只是轻微加分。

它主要看三类信息。

第一类是己方进攻：

```text
我下这里能不能成五？
我下这里能不能形成四？
我下这里能不能形成三？
```

第二类是防守价值：

```text
如果对手下这里，会不会赢？
如果对手下这里，会不会形成强威胁？
```

第三类是位置偏好：

```text
越靠近中心，略微加一点分。
```

为什么要同时看进攻和防守？因为好棋不一定是“我马上变强”的点，也可能是“我必须挡住对方”的点。比如对方已经有四连，你再去自己进攻就可能下一手直接输。

## 6. 必胜和必防优先级

在正式搜索之前，`chooseMove` 会先扫一遍候选点：

1. 如果己方有一步能赢，直接下。
2. 否则如果对手有一步能赢，直接挡。

这是一个很朴素但很有效的优化。

如果一个点下去已经赢了，就没必要搜索未来 4 层。如果对手下一步就能赢，也不应该把希望寄托在评估函数上，必须优先堵住。

这个设计也保护了随机版。随机版不会在“一步成五”和“必须挡五”的场景下乱选，它只在没有这种强制战术时才随机。

## 7. 局面评估函数

当搜索到指定深度，或者时间快用完，程序需要判断当前局面“看起来谁更好”。这个工作由 `evaluateBoard` 完成：

```cpp
static int evaluateBoard(int color)
```

它扫描棋盘上所有长度为 5 的窗口。比如横向连续 5 格、纵向连续 5 格、两条斜线方向连续 5 格。

每个窗口统计：

```text
里面有几个黑棋？
里面有几个白棋？
里面有几个空点？
```

然后给分：

```text
己方五连：极高正分
对手五连：极高负分
己方四子一空：高正分
对手四子一空：高负分
己方三子两空：中等正分
对手三子两空：中等负分
```

如果一个 5 格窗口里黑白都有棋，它基本不可能成为任何一方的五连，所以这个窗口记 0 分。

可以把评估函数理解成一个“局面温度计”。它不保证绝对准确，但能快速告诉搜索算法：这个局面大概是我更舒服，还是对方更危险。

## 8. Alpha-Beta 搜索

核心搜索函数是：

```cpp
static int alphaBeta(...)
```

它基于极大极小搜索。简单说：

- 我方回合，会选择让我分数最高的走法。
- 对方回合，会选择让我分数最低的走法。

如果把棋局想象成一棵树，每一层是一方下一步，那么 AI 就是在这棵树里往下看几层，估计每个选择会带来什么后果。

### 8.1 极大极小的直觉

假设我有两个选择：

```text
A：如果对手最佳应对，我最后 +100
B：如果对手最佳应对，我最后 +50
```

我会选 A。

但对手不是配合我的，他也会找我的麻烦。所以评价一个走法时，不能只看“我下一步多爽”，还要看“对手如果下得最好，我还能不能占优”。

### 8.2 Alpha-Beta 剪枝

纯极大极小会搜索很多没必要的分支。Alpha-beta 剪枝的作用是：如果已经知道某个分支不可能影响最终选择，就不再继续算它。

通俗点说，假设我已经找到一个能拿 100 分的走法，现在分析另一个走法时发现，对手只要一个应对就能让我最多拿 20 分，那这个走法就不可能更好，后面不用看了。

剪枝不会改变搜索结果，只是少算很多无用分支。候选点排序越好，越早看到强手，剪枝效果越明显。

## 9. 迭代加深和时间控制

程序不是直接固定搜 4 层或 5 层，而是从浅到深：

```text
先搜 1 层
再搜 2 层
再搜 3 层
直到达到最大深度或时间用完
```

这种方法叫迭代加深。

它的好处是：如果时间突然不够，至少已经有一个浅层搜索得到的可用答案，而不是算到一半什么都没有。

不同 exe 的时间和深度由编译宏决定：

| 档位 | 时间 | 最大深度 | 根候选 | 深层/浅层候选 |
| --- | --- | --- | --- | --- |
| 超级 | `2800ms` | `5` | `40` | `18 / 26` |
| 中等 | `1500ms` | `4` | `30` | `14 / 20` |
| 快速 | `900ms` | `4` | `24` | `10 / 16` |
| 超快速 | `350ms` | `3` | `18` | `8 / 12` |

这些参数体现了速度和棋力的取舍。候选越多、深度越深，AI 看得越细，但耗时也越大。

## 10. 置换表和 Zobrist 哈希

搜索时，经常会通过不同顺序到达同一个局面。例如：

```text
我先下 A，对手下 B
我先下 C，对手下 D
```

在复杂搜索中，某些局面可能重复出现。如果每次都重新搜索，会浪费很多时间。

程序用 Zobrist 哈希给棋盘生成一个 `uint64_t` 的局面编号：

```cpp
static uint64_t zobrist[N][N][3];
static uint64_t currentHash;
```

每个位置、每种棋子都有一个随机数。落子时把对应随机数异或进 `currentHash`，撤子时再异或一次就能恢复。这个特性很适合搜索，因为 alpha-beta 经常需要“下一步、撤回来、再试下一步”。

置换表 `transTable` 用这个哈希作为 key，保存已经搜索过的结果：

```cpp
depth
value
flag
```

其中 `flag` 表示这个值是精确值、下界还是上界。这样下次遇到同样局面时，如果之前搜索深度足够，就可以复用结果。

## 11. 黑方禁手处理

本项目规则要求黑方有禁手：

- 三三禁手
- 四四禁手
- 长连禁手

核心函数是：

```cpp
static bool isForbiddenBlackMove(int r, int c)
```

注意：禁手只在黑方主动落子时判断。白方下一步导致黑方棋形变成禁手，不算黑方犯规。

### 11.1 长连

长连指黑方落子后某个方向连续超过 5 个黑棋。程序通过 `hasOverlineAt` 判断。

但是有一个特殊规则：如果黑方本手形成恰好五连，即使同时有禁手形状，也按黑方胜利。本项目用 `hasExactFiveAt` 先判断是否恰好五连，如果是，就不再把它当禁手。

### 11.2 四四

四四可以理解为：黑方一手落下后，同时形成两个方向上的“四”的威胁。程序的做法是：对每个方向，尝试找这个方向上有没有一个后续点能形成恰好五连。如果有，这个方向算一个“四”。如果两个或更多方向都有这种威胁，就是四四禁手。

### 11.3 三三

三三比四四更难，因为“活三”有直三和跳三等形式。程序在落子附近取一条长度 11 的局部线段，然后匹配一些常见开放三形状：

```text
.XXX.
.XX.X.
.X.XX.
.XX..X.
.X.X.X.
```

这里 `.` 表示空点，`X` 表示黑棋。棋盘边界被当成白棋阻挡。若两个方向都出现开放三，就判为三三禁手。

这种方法不是职业级禁手解析器，但对课程项目和普通对局已经能覆盖常见形状，而且逻辑清楚、速度快。

## 12. 随机版为什么能避免每盘一样

默认版本是确定性的。原因有三个：

1. Zobrist 随机数使用固定种子。
2. 候选点排序稳定。
3. 同一局面搜索结果相同。

这样方便调试，但会导致同样输入每盘都一样。

随机版通过 `RANDOM_AI` 宏启用，主要做两件事。

第一，首手从中心附近随机选：

```text
中心 3x3 范围内的合法点
```

第二，搜索结束后不一定只拿唯一最高分，而是收集与最高分接近的一组候选点，然后随机选一个。也就是说，它不是“随便下”，而是在几步差不多好的棋里面换着下。

如果需要复现，设置：

```powershell
$env:GOMOKU_SEED=12345
```

同样 seed 加同样输入，就会得到同样输出。删除环境变量后，又会恢复每次随机。

## 13. 显示棋盘版本

显示版通过 `SHOW_BOARD` 宏启用。它不会改变 AI 算法，只是在每次自己落子后把棋盘打印到 `stderr`。

为什么不用 `stdout` 打印棋盘？因为裁判程序通常从 `stdout` 读取坐标。如果棋盘也打印到 `stdout`，裁判可能读错。使用 `stderr` 可以让人在控制台看到棋盘，又尽量不破坏坐标协议。

棋盘字符：

```text
X = 黑棋
O = 白棋
. = 空点
```

## 14. 当前 AI 的强项

这个 AI 的优点主要有：

- 协议简单，适合接裁判程序。
- 黑方禁手规则有独立处理。
- 先处理一步胜和一步防，避免明显低级错误。
- 候选点生成减少大量无意义搜索。
- alpha-beta 加排序，搜索效率比暴力极大极小高很多。
- 多档位 exe 适合不同时间限制。
- 随机版能避免每盘完全重复。

## 15. 当前 AI 的局限

它也不是职业级五子棋 AI，主要局限有：

- 评估函数仍然偏简单，主要看长度 5 的窗口。
- 三三禁手使用模式匹配，覆盖常见形状，但不是完整形式化解析。
- 没有专门的威胁空间搜索，也没有证明级必胜搜索。
- 搜索深度有限，长线布局和复杂交换可能看不远。
- 候选点只看已有棋子附近，极少数远距离战略点可能被忽略。

这些局限不是 bug，而是速度、代码复杂度和课程项目可解释性之间的取舍。

## 16. 可以怎样继续升级

如果要继续提升棋力，可以考虑：

- 更精细的棋形评估，比如区分活四、冲四、眠三、活三、跳三。
- 更完整的禁手检测，尤其是三三和四四的严谨定义。
- 威胁空间搜索，专门分析连续强迫手。
- 开局库，让前几十手更接近成熟定式。
- 更强的置换表策略，比如限制大小、替换策略、保存最佳着法。
- 使用位棋盘或增量评估，减少每次评估全盘扫描的开销。
- 对不同档位使用不同评估细节，强档更复杂，快档更简单。

## 17. 一句话总结

这个 AI 的核心思路是：先用候选点生成把问题变小，再用局部评分把好棋排前面，然后用 alpha-beta 往后看几步，最后用评估函数判断局面好坏。禁手、随机版、显示版和速度档位都是围绕这个核心搜索框架扩展出来的。
