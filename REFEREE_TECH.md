# 五子棋图形化裁判技术说明

本文介绍 `gomoku_referee.py` 的设计和实现。它不是一个下棋 AI，而是一个“裁判程序”：负责启动双方程序、收发坐标、维护棋盘、判断胜负和犯规、显示界面、记录日志、统计比分，并在一盘结束后支持复盘。

读这份文档不需要非常深入的 Python GUI 经验，但最好已经知道一些基础概念，比如标准输入输出、子进程、线程、队列、事件循环、二维数组。文档会尽量用通俗的话解释：先说明为什么要这样做，再说明代码大概怎么实现。

## 1. 为什么需要独立裁判

在五子棋程序对战中，最容易出问题的地方不是“AI 会不会下”，而是“双方到底有没有按同一套规则下”。如果只让两个 exe 互相输入输出坐标，会出现几个麻烦：

- 有的程序只输出坐标，有的程序会顺手输出提示语、中文说明或棋盘。
- 有的程序可能没有实现完整禁手规则。
- 有的程序可能输出了越界坐标、已经有棋子的坐标，或者干脆超时不输出。
- 有的同学可能只写了白方或只写了黑方，黑方第一手就无法正常开始。
- 人想和 exe 对打时，需要一个界面把“点击棋盘”转换成协议里的 `row col`。

所以裁判程序的核心思想是：双方 exe 只负责“给出下一步坐标”，至于这一步是否合法、谁赢、谁超时、棋盘怎么显示，都由裁判统一决定。这样对战结果更公平，也更容易调试。

当前裁判文件是：

```text
gomoku_referee.py
```

它使用 Python 标准库里的 `tkinter` 做界面，不依赖第三方库。直接运行即可：

```powershell
python gomoku_referee.py
```

## 2. 基本运行方式

运行后会打开一个窗口，左边是 38 路棋盘，右边是控制区、日志和累计比分。顶部可以选择对局模式：

- `黑 exe vs 白 exe`：两个外部程序自动对打。
- `用户黑 vs exe 白`：用户点击棋盘执黑，白方由 exe 下。
- `exe 黑 vs 用户白`：黑方由 exe 下，用户点击棋盘执白。
- `用户 vs 用户`：双方都由用户点击棋盘。

如果某一方是 exe，就需要在界面里选择对应的可执行文件。例如黑方选择 `black.exe`，白方选择 `white.exe`。点击“开始本盘”后，裁判会根据模式启动需要的进程，并从黑方开始推进对局。

界面还有几个常用按钮：

- “开始本盘”：清空棋盘，按当前模式开始新的一盘。
- “停止本盘”：终止当前对局和已启动的 exe。
- “下一盘”：保留累计比分，重新开始一盘。
- “重置比分”：把运行期间累计的胜负统计清零。

一盘结束后，棋盘下面的复盘进度条会启用。把进度条拖到 `0`，显示空棋盘；拖到 `n`，显示前 `n` 手。复盘只改变显示，不会改变真实对局结果，也不会重复统计比分。

## 3. 坐标和棋盘表示

外部协议使用 `1..38` 的坐标，格式是：

```text
row col
```

例如：

```text
19 19
```

这表示第 19 行第 19 列。这个格式是给 exe 看、给日志看、给用户看的。

程序内部使用 Python 二维列表保存棋盘，坐标是 `0-based`，也就是范围 `0..37`。所以外部的 `(19, 19)` 在内部是 `(18, 18)`。

代码里的核心常量是：

```python
BOARD_SIZE = 38
EMPTY = 0
BLACK = 1
WHITE = 2
```

棋盘数据结构大致是：

```python
board[row][col]
```

其中：

- `0` 表示空位。
- `1` 表示黑棋。
- `2` 表示白棋。

坐标转换由 `to_external(r, c)` 完成：

```python
def to_external(r, c):
    return r + 1, c + 1
```

这个函数看起来很小，但很重要。只要程序内部和外部混用坐标，就很容易出现“界面点的是 19 19，发给 exe 却变成 18 18”的错误。裁判统一规定：内部一律 0-based，对外一律 1-based。

## 4. 整体架构

`gomoku_referee.py` 是单文件程序，但内部按职责分成四层：

```text
BoardRules      只管棋盘和规则
EngineProcess   只管 exe 进程和输入输出
GameController  只管一盘棋如何推进
GomokuApp       只管 Tkinter 界面
```

这四层可以理解成：

- `BoardRules` 是规则老师：问它“这步能不能下”“有没有赢”“是不是禁手”。
- `EngineProcess` 是翻译和传话人：负责启动 exe，把坐标写给它，再读取它输出的内容。
- `GameController` 是比赛组织者：决定现在轮到谁、要等哪个 exe、超时怎么办、什么时候结束。
- `GomokuApp` 是前台界面：画棋盘、显示日志、响应按钮和鼠标点击。

这样的分层有一个好处：如果以后想换 GUI，比如换成 PySide6 或网页界面，规则和进程通信不用大改；如果以后想改禁手规则，也不需要碰界面绘制代码。

## 5. `BoardRules`：规则判定层

`BoardRules` 负责维护棋盘，并回答所有规则问题。它不关心这一步来自用户还是 exe，也不关心 GUI 怎么显示。

### 5.1 基础操作

它提供几个基础方法：

- `inside(r, c)`：判断坐标是否在棋盘内。
- `is_empty(r, c)`：判断某格是否为空。
- `put(r, c, color)`：放下一颗棋。
- `remove(r, c)`：移走一颗棋。
- `count_line(r, c, dr, dc, color)`：从某点沿某个方向统计连续棋子数。

五子棋只需要检查四个方向：

```python
DIRS = ((1, 0), (0, 1), (1, 1), (1, -1))
```

它们分别表示：

- 竖向。
- 横向。
- 左上到右下的斜线。
- 右上到左下的斜线。

为什么只要四个方向？因为每个方向都会同时向正反两边数。例如横向 `(0, 1)` 已经能同时检查左边和右边，不需要再写 `(0, -1)`。

### 5.2 黑白胜利规则不同

本项目采用的规则是：

- 黑方胜利：必须形成“恰好五连”。
- 白方胜利：形成“五连或更长”都算胜。

所以代码里分成两个判断：

```python
has_exact_five_at(...)
has_five_or_more_at(...)
```

最后统一由 `wins_at(r, c, color)` 调用：

```python
if color == BLACK:
    return has_exact_five_at(...)
return has_five_or_more_at(...)
```

举个例子：黑方横向连了 6 个，这叫长连。黑方不能靠长连赢，除非同一手在另一个方向还同时形成了恰好五连。白方没有这个限制，白方连 6 个也算胜。

### 5.3 黑方禁手

黑方主动落子时需要检查：

- 长连禁手。
- 四四禁手。
- 三三禁手。

但有一个很关键的优先级：

> 黑方如果本手形成恰好五连，即使同时出现禁手形状，也按黑方胜利处理。

所以 `forbidden_reason_for_black_move(r, c)` 的顺序是：

1. 先检查越界和已有棋子。
2. 临时把黑棋放到棋盘上。
3. 如果形成恰好五连，直接返回 `None`，表示不是禁手。
4. 再检查长连、四四、三三。
5. 最后把临时棋子恢复为空。

伪代码可以这样理解：

```text
如果坐标不合法：返回原因
临时放黑棋
如果恰好五连：允许
如果长连：禁止
如果两个方向成四：禁止
如果两个方向成活三：禁止
否则允许
撤销临时黑棋
```

注意这里说的是“黑方主动落子”。如果白方下一手导致棋盘上黑方某些形状看起来像禁手，裁判不会追溯判黑方犯规。这就是“黑方被动形成禁手不判犯规”。

### 5.4 四四和三三怎么近似判断

四四禁手的判断思路是：黑方落下这一手后，看四个方向里，有多少个方向存在“下一步可以形成恰好五连”的点。如果这样的方向达到两个，就认为是四四。

三三禁手的判断思路是：黑方落下这一手后，围绕这颗新棋，在每个方向取一段局部线，然后匹配几种常见活三形状，例如：

```text
空 黑 黑 黑 空
空 黑 黑 空 黑 空
空 黑 空 黑 黑 空
```

如果活三方向达到两个，就认为是三三。

这里的实现是规则裁判常见的“局部模式匹配”写法。它比完整的专业禁手引擎短很多，也更容易读懂；代价是如果要覆盖所有竞赛级禁手细节，未来可能还需要继续扩展模式。但对于当前项目文档规定的规则测试和常见局面，它的逻辑已经能独立完成裁判工作。

## 6. `EngineProcess`：exe 通信层

`EngineProcess` 只负责一件事：把一个 exe 当成外部选手来管理。

启动 exe 使用 `subprocess.Popen`：

```python
subprocess.Popen(
    [path],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True,
    encoding="utf-8",
    errors="replace",
)
```

这几个参数的含义是：

- `stdin=PIPE`：裁判可以给 exe 写入对手坐标。
- `stdout=PIPE`：裁判可以读取 exe 输出的落子。
- `stderr=PIPE`：裁判可以读取 exe 的提示、棋盘或错误信息。
- `text=True`：按文本处理，而不是按字节处理。
- `errors="replace"`：遇到无法解码的字符时替换掉，不让程序直接崩溃。

### 6.1 为什么要分线程读取输出

GUI 程序最怕“卡住”。如果 Tkinter 主线程直接调用 `stdout.readline()`，而 exe 此时没有输出，整个窗口就会停在那里，按钮点不了，棋盘也不会刷新。

因此裁判采用后台线程读取：

```text
stdout 读取线程 -> stdout_queue
stderr 读取线程 -> stderr_queue
Tkinter 主线程每 100ms 从队列取走已有内容
```

线程只负责“读到什么就放进队列”，不直接修改界面。界面更新仍由 Tkinter 主线程完成。这样做更安全，因为 Tkinter 本身并不适合在多个线程里同时操作控件。

### 6.2 stdout 和 stderr 分工

裁判只从 `stdout` 解析落子坐标。`stderr` 会显示到日志里，但不会参与坐标解析。

这样设计是有原因的：

- 正常协议要求 stdout 输出坐标。
- 可视化版程序可能把棋盘输出到 stderr。
- 如果 stderr 也参与解析，棋盘上的行列数字很容易被误当成落子。

所以最稳的约定是：stdout 是正式答题纸，stderr 是旁边的草稿纸。裁判只从正式答题纸里找坐标。

### 6.3 给 exe 发送坐标

当某一方成功落子后，如果下一方是 exe，裁判会把刚刚的坐标写入下一方进程：

```text
row col\n
```

例如黑方下了内部 `(18, 18)`，外部坐标就是：

```text
19 19
```

`EngineProcess.write_move()` 会完成转换、写入和 `flush()`。`flush()` 很重要，因为没有它，文本可能还停在缓冲区里，exe 就收不到输入。

## 7. `GameController`：对局流程层

`GameController` 是整盘棋的核心调度器。它知道：

- 当前轮到谁。
- 双方是用户还是 exe。
- 当前是否正在等某个 exe 输出。
- 一盘棋是否已经结束。
- 棋谱历史和累计比分是多少。

它不直接画棋盘，只会调用 `app.draw_board()`、`app.log()`、`app.set_status()` 之类的方法通知界面更新。

### 7.1 开始一盘棋

点击“开始本盘”后，流程大致是：

```text
停止上一盘残留进程
清空棋盘
清空本盘历史
根据模式确定双方身份
启动需要的 exe
设置当前方为黑方
进入回合推进
```

如果模式是 `exe_exe`，黑白双方都会启动 exe。如果是 `用户黑 vs exe 白`，只启动白方 exe。如果是 `用户 vs 用户`，不启动任何外部程序。

### 7.2 回合推进

每个回合只有两种情况：

```text
当前方是用户 -> 等待用户点击棋盘
当前方是 exe  -> 等待 stdout 输出坐标
```

用户点击棋盘时，GUI 会把鼠标位置转成棋盘坐标，然后调用：

```python
controller.human_move(r, c)
```

exe 回合则由 `tick()` 周期性检查 stdout 队列。如果读到了内容，就尝试解析坐标；如果解析成功，就调用统一的 `apply_move(...)`。

这一点很重要：不管落子来自用户、exe，还是黑方首手超时后的裁判随机落子，最后都会走 `apply_move()`。这样规则判断、胜负判断、棋谱记录、界面刷新都不会分散到很多地方。

### 7.3 一步棋如何落到棋盘上

`apply_move()` 的职责可以分成几步：

```text
确认现在确实轮到这个颜色
调用 BoardRules.validate_move 检查合法性
非法则结束对局并判负
合法则真正 put 到棋盘
记录 MoveRecord
刷新棋盘和复盘范围
检查是否胜利
检查是否平局
把本步坐标发送给下一方 exe
切换 current 到下一方
继续推进下一回合
```

这里有一个小细节：如果黑方下出三三、四四或长连，`validate_move()` 会返回不合法，`GameController` 会把它单独记为“黑方犯规负”。如果是越界、重复落子、超时、进程退出等，则记到“非法/超时负”。

## 8. 模糊坐标解析

理想情况下，exe 每次只输出一行：

```text
19 19
```

但实际同学写的程序可能会输出：

```text
我选择落子：19 19
```

或者：

```text
AI thinking...
最终坐标 19 19
```

为了兼容这些情况，裁判使用正则表达式从 stdout 中提取整数：

```python
re.findall(r"-?\d+", text)
```

然后把相邻两个整数看成一组候选坐标。比如输出里有这些数字：

```text
19 19 20 20
```

裁判会依次尝试：

```text
(19, 19)
(19, 20)
(20, 20)
```

找到第一组满足条件的坐标后就采用：

- 行列都在 `1..38`。
- 该位置当前为空。

按照当前设计，裁判采用“第一组当前可用合法坐标”。如果这组坐标是黑方禁手，裁判不会继续找后面的坐标，而是认为该 exe 确实选择了这个点，然后按规则判黑方犯规。这是为了避免程序在输出一堆数字时“碰巧后面还有一个合法点”导致裁判替它改棋。

需要注意：如果程序把完整棋盘、行列号、很多调试数字都打印到 stdout，依然可能干扰解析。更推荐的做法是：

- stdout 只输出最终坐标。
- 调试信息、棋盘、中文说明放到 stderr。

裁判已经支持读取 stderr 并写入日志。

## 9. 超时机制

超时是裁判公平性的另一个关键点。否则某个 exe 一直不输出，比赛就无法继续。

当前有两种超时：

### 9.1 黑方第一手 5 秒特殊超时

黑方是先手，按协议应该启动后立刻输出第一手。但考虑到有些同学可能没有实现黑方程序，或者黑方程序启动后不会主动输出，裁判给了一个特殊兜底：

- 如果黑方是 exe。
- 当前是整盘第一手。
- 等待 5 秒仍没有坐标。

裁判会在中心 9 格里随机选一个空点作为黑方第一手。中心 9 格内部坐标是 `17..19`，对应外部坐标是 `18..20`。

也就是说可能随机到：

```text
18 18
18 19
18 20
19 18
19 19
19 20
20 18
20 19
20 20
```

这一步来源会在日志里显示为“裁判随机”，不判黑方负。之后比赛继续进行，白方 exe 或用户会收到这一步坐标。

### 9.2 普通回合 10 秒超时

除黑方第一手特殊情况外，每个 exe 回合超时时间是 10 秒。如果 10 秒内没有解析到可用坐标，则当前方判负。

这个机制覆盖几种情况：

- exe 死循环。
- exe 等输入等错了。
- exe 输出了内容，但里面没有可用坐标。
- exe 输出太慢。

如果 exe 直接退出，裁判也会判该方负，并把原因写进日志。

## 10. `GomokuApp`：界面层

`GomokuApp` 负责 Tkinter 界面。它创建窗口、按钮、棋盘、日志框、比分框和复盘进度条。

### 10.1 棋盘绘制

棋盘使用 `tk.Canvas` 绘制。核心参数是：

```python
self.margin = 24
self.cell = 17
```

`margin` 是棋盘边缘留白，`cell` 是相邻交点之间的像素距离。因为是 38 路棋盘，一共有 38 个交点，所以画线长度按 `BOARD_SIZE - 1` 个间隔计算。

棋子用圆形绘制：

- 黑棋：深色圆。
- 白棋：浅色圆。
- 最后一步：用红色小方框标记。

鼠标点击时，界面会把像素坐标转换成棋盘坐标。转换方式不是简单向下取整，而是 `round()` 到最近交点，并检查点击位置是否离交点足够近。这样用户不用点得特别准。

### 10.2 Tkinter 事件循环

Tkinter 程序不能像命令行程序那样写一个死循环一直等输入。它靠主循环处理事件：

```python
root.mainloop()
```

为了让裁判持续检查 exe 输出，程序使用：

```python
self.root.after(100, self.periodic_tick)
```

意思是每隔约 100ms 调用一次 `periodic_tick()`。`periodic_tick()` 再调用 `controller.tick()`，检查 stdout、stderr、超时和进程状态。

这是一种 GUI 程序里常见的写法：不阻塞主线程，而是定期“看一眼有没有新事情”。所以窗口能保持响应，日志也能及时刷新。

### 10.3 日志和状态

日志框记录关键事件，例如：

- exe 启动路径。
- 收到 stdout 内容。
- stderr 输出。
- 发送给对方的坐标。
- 用户或 exe 的落子。
- 超时、非法落子、禁手、胜负结果。

状态栏则显示当前最重要的一句话，例如“轮到黑方，请点击棋盘落子”或“等待白方 exe 输出，剩余 7.3s”。

日志适合事后排查，状态栏适合当前操作。

## 11. 棋谱、复盘和累计比分

每一步棋都会记录为 `MoveRecord`：

```python
@dataclass
class MoveRecord:
    color: int
    row: int
    col: int
    source: str
    raw: str = ""
    random_first: bool = False
    timestamp: float = 0.0
```

它保存：

- 哪一方下的。
- 内部坐标。
- 来源是用户、exe，还是裁判随机。
- exe 原始输出内容。
- 是否为黑方首手随机。
- 时间戳。

这些记录组成 `move_history`。复盘时，程序不会修改真实棋盘，而是重新创建一个空棋盘，然后把前 `n` 手依次放上去：

```text
空棋盘
放第 1 手
放第 2 手
...
放第 n 手
显示这个临时棋盘
```

所以复盘是只读展示，不会影响对局状态。

累计比分保存在 `stats` 字典里，包括：

- 总局数。
- 黑胜。
- 白胜。
- 平局。
- 黑方犯规负。
- 黑方非法/超时负。
- 白方非法/超时负。

这些数据只在程序运行期间存在。关闭程序后不会自动保存。如果以后需要长期统计，可以把它扩展为 CSV、JSON 或数据库记录。

## 12. 一盘 exe vs exe 的完整流程

下面用“黑 exe vs 白 exe”举一个完整流程：

```text
1. 用户选择 black.exe 和 white.exe。
2. 用户点击开始本盘。
3. 裁判启动黑方进程和白方进程。
4. 当前方设为黑方。
5. 黑方是 exe，裁判开始等待黑方 stdout，首手超时为 5 秒。
6. 黑方输出 19 19。
7. 裁判解析坐标，检查合法性，放下黑棋。
8. 裁判检查黑方是否胜利或犯规。
9. 裁判把 19 19 写入白方 stdin。
10. 当前方切换为白方。
11. 白方是 exe，裁判等待白方 stdout，普通超时为 10 秒。
12. 白方输出 20 18。
13. 裁判解析、检查、落子、刷新界面。
14. 裁判把 20 18 写入黑方 stdin。
15. 重复以上流程，直到胜利、犯规、非法、超时或平局。
16. 对局结束，终止 exe，更新比分，启用复盘。
```

用户参与时，流程几乎一样，只是“等待 stdout”换成“等待鼠标点击”。

## 13. 常见问题和排错

### 13.1 exe 没有输出

如果黑方第一手没有输出，5 秒后裁判会随机中心落子。普通回合 10 秒没有输出则判负。

排查时可以先在命令行单独运行：

```powershell
.\black.exe
```

黑方程序应该启动后直接输出一个坐标。白方程序则应该先等待输入：

```powershell
@('19 19') | .\white.exe
```

### 13.2 exe 输出了中文，能不能识别

可以，只要 stdout 里包含可用坐标。例如：

```text
我方落子：19 19
```

裁判会提取整数并解析。更推荐最终 stdout 尽量只放坐标，其他说明放 stderr。

### 13.3 输出棋盘导致坐标解析错

如果把棋盘、行号、列号都输出到 stdout，裁判可能先抓到棋盘上的数字，而不是最终落子。可视化输出建议放到 stderr。

项目里的 `black_show.exe`、`white_show.exe` 就是这种设计：坐标仍在 stdout，棋盘在 stderr。

### 13.4 为什么我的黑方长连不是胜利

这是规则决定的。黑方胜利按“恰好五连”判断，长连属于禁手。如果黑方下完后只有六连或更长，没有任何方向恰好五连，就不是胜利，而是犯规。

白方不同。白方五连或更长都算胜。

### 13.5 为什么已经形成禁手没有判

裁判只检查“黑方主动落子”造成的禁手。如果白方落子后，棋盘上出现了某种黑方禁手形状，不会追溯判黑方犯规。这叫黑方被动形成禁手不判。

### 13.6 窗口看起来卡住

正常情况下，裁判不会因为等待 exe 而卡住，因为 stdout/stderr 在后台线程读取，主线程每 100ms 轮询。如果窗口真的无响应，可能是系统层面的进程问题，或代码被修改后在主线程里加入了阻塞调用。

## 14. 可扩展方向

当前裁判已经能满足基本对战和观察需求，但后续还可以继续增强：

- 导出棋谱：把 `move_history` 保存为 JSON、CSV 或自定义棋谱格式。
- 保存累计统计：关闭程序后仍保留历史胜负。
- 批量对战：指定两个 exe 后自动连续下 100 盘并统计胜率。
- 打包 exe：用 PyInstaller 把 `gomoku_referee.py` 打包成 Windows 可执行文件。
- 更严格禁手引擎：扩展活三、冲四、禁手模式，让规则更接近专业比赛裁判。
- 可配置时间：把 5 秒首手超时、10 秒普通超时做成界面输入项。
- 可配置棋盘大小：如果以后不只做 38 路，可以把 `BOARD_SIZE` 外置成配置。
- 对局导入：从日志或棋谱文件重新加载一盘棋进行复盘。

## 15. 小结

`gomoku_referee.py` 的重点不是算法有多复杂，而是把几个容易混在一起的问题拆开：

- 棋盘规则交给 `BoardRules`。
- exe 通信交给 `EngineProcess`。
- 对局推进交给 `GameController`。
- 界面展示交给 `GomokuApp`。

这种拆法让程序比较容易理解，也方便后续维护。你可以把它看成一个小型比赛平台：AI 只需要按协议输出坐标，裁判负责保证比赛按统一规则进行。
